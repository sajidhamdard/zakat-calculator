JavaScript Coding Guidelines & Style Guide (2026)

Naming conventions, functions, variables, error handling, scoping, formatting, and real project examples

This comprehensive JavaScript coding guidelines and style guide covers naming conventions, functions, variables, statements, error handling, scoping, formatting, comments, real project folder structure examples, common JavaScript coding mistakes, and coding standards frequently asked in JavaScript interview questions. Each section includes practical code examples you can apply immediately in your projects. If you want to level up your modern JavaScript skills, also explore our guide on JavaScript best practices.

Introduction

▼

Writing clean, consistent JavaScript is one of the most important skills for any developer — whether you are working alone or in a large team. Coding guidelines ensure everyone writes code the same way, making it easier to read, review, and maintain over time.

These guidelines follow modern ECMAScript standards (ES6+) and are designed to be practical: every rule is explained with a reason and illustrated with a before/after code example.

💡 Tip: Enforce these guidelines automatically using ESLint and Prettier in your project.

Naming Conventions

▼

Consistent naming makes code self-documenting and reduces the need for comments.

Variables and Functions — camelCase

// ✅ Good
const userAge = 25;
function calculateTotalPrice(items) { /* … */ }

// ❌ Bad
const UserAge = 25;
function Calculate_Total_Price(items) { /* … */ }

Classes and Constructors — PascalCase

// ✅ Good
class ShoppingCart { /* … */ }
const cart = new ShoppingCart();

// ❌ Bad
class shoppingcart { /* … */ }

Constants — UPPER_SNAKE_CASE

// ✅ Good
const MAX_RETRY_ATTEMPTS = 3;
const API_BASE_URL = 'https://api.example.com';

// ❌ Bad
const maxRetryAttempts = 3;

Boolean Variables — use is / has / can prefix

// ✅ Good
const isLoggedIn = true;
const hasPermission = false;
const canEdit = user.role === 'admin';

// ❌ Bad
const loggedIn = true;
const permission = false;

Functions

▼

Prefer Arrow Functions for Callbacks

// ✅ Good
const doubled = numbers.map(n => n * 2);

// ❌ Avoid (verbose)
const doubled = numbers.map(function(n) { return n * 2; });

Keep Functions Small and Single-Purpose

// ✅ Good – one responsibility
function formatCurrency(amount, currency = 'USD') {
  return new Intl.NumberFormat('en-US', { style: 'currency', currency }).format(amount);
}

// ❌ Bad – does too much
function processUser(user) {
  // validates, formats, saves, and sends email — too much!
}

Use Default Parameters

// ✅ Good
function greet(name = 'Guest') {
  return `Hello, ${name}!`;
}

// ❌ Bad
function greet(name) {
  name = name || 'Guest';
  return `Hello, ${name}!`;
}

Variables

▼

Prefer const, then let — never var

// ✅ Good
const PI = 3.14159;
let counter = 0;

// ❌ Bad
var PI = 3.14159;
var counter = 0;

Declare Variables at the Top of Their Scope

// ✅ Good
function processOrder(order) {
  const { id, items, discount } = order;
  const total = calculateTotal(items, discount);
  return { id, total };
}

Use Destructuring

// ✅ Good
const { firstName, lastName, email } = user;
const [first, ...rest] = items;

// ❌ Bad
const firstName = user.firstName;
const lastName = user.lastName;

Statements

▼

One Statement Per Line

// ✅ Good
const a = 1;
const b = 2;

// ❌ Bad
const a = 1; const b = 2;

Use Strict Equality (===)

// ✅ Good
if (value === null) { /* … */ }
if (count === 0) { /* … */ }

// ❌ Bad – loose equality causes unexpected type coercion
if (value == null) { /* … */ }
if (count == 0) { /* … */ }

Prefer Early Returns to Reduce Nesting

// ✅ Good
function getDiscount(user) {
  if (!user) return 0;
  if (!user.isPremium) return 0;
  return 0.15;
}

// ❌ Bad – deeply nested
function getDiscount(user) {
  if (user) {
    if (user.isPremium) {
      return 0.15;
    }
  }
  return 0;
}

Error Handling

▼

Always Handle Promise Rejections

// ✅ Good — async/await with try/catch
async function fetchUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);
    if (!response.ok) throw new Error(`HTTP error: ${response.status}`);
    return await response.json();
  } catch (error) {
    console.error('Failed to fetch user:', error);
    throw error;
  }
}

// ❌ Bad — unhandled rejection
fetch('/api/users/1').then(res => res.json());

Use Custom Error Classes for Domain Errors

class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = 'ValidationError';
    this.field = field;
  }
}

throw new ValidationError('Email is required', 'email');

Scoping

▼

Limit Global Variables

// ✅ Good — encapsulate in a module or IIFE
const AppConfig = (() => {
  const _apiKey = 'secret';
  return { getApiKey: () => _apiKey };
})();

// ❌ Bad — pollutes global scope
var apiKey = 'secret';

Understand Block Scoping with let / const

// ✅ Good
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); // logs 0, 1, 2
}

// ❌ Bad
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); // logs 3, 3, 3
}

Formatting

▼

Use 2-Space Indentation

function add(a, b) {
  if (a === undefined) {
    return b;
  }
  return a + b;
}

Use Template Literals for String Interpolation

// ✅ Good
const message = `Hello, ${user.name}! You have ${count} messages.`;

// ❌ Bad
const message = 'Hello, ' + user.name + '! You have ' + count + ' messages.';

Trailing Commas in Multi-Line Collections

// ✅ Good — makes diffs cleaner
const config = {
  host: 'localhost',
  port: 3000,
  debug: true,   // trailing comma
};

const items = [
  'apple',
  'banana',
  'cherry',   // trailing comma
];

Comments

▼

Comment the Why, Not the What

// ✅ Good — explains the reason
// Multiply by 1000 because the API returns seconds, but Date expects ms
const date = new Date(timestamp * 1000);

// ❌ Bad — obvious from the code itself
// multiply timestamp by 1000
const date = new Date(timestamp * 1000);

Use JSDoc for Public APIs

/**
 * Calculates the Zakat amount due on a given wealth.
 * @param {number} wealth - Total zakatable assets in currency units.
 * @param {number} [rate=0.025] - Zakat rate (default 2.5%).
 * @returns {number} The Zakat amount due.
 */
function calculateZakat(wealth, rate = 0.025) {
  return wealth * rate;
}

Common JavaScript Mistakes

▼

1. Mutating Objects Passed as Arguments

// ❌ Bad — mutates the original object
function addFullName(user) {
  user.fullName = `${user.first} ${user.last}`;
  return user;
}

// ✅ Good — returns a new object
function addFullName(user) {
  return { ...user, fullName: `${user.first} ${user.last}` };
}

2. Forgetting await in Async Functions

// ❌ Bad — returns a Promise, not the resolved value
async function getUser(id) {
  const user = fetch(`/api/users/${id}`); // missing await
  return user;
}

// ✅ Good
async function getUser(id) {
  const response = await fetch(`/api/users/${id}`);
  return response.json();
}

3. Using == Instead of ===

// ❌ Bad — 0 == '' is true, which is surprising!
if (value == '') { /* … */ }

// ✅ Good
if (value === '') { /* … */ }

4. Not Handling Edge Cases in Array Methods

// ❌ Bad — crashes if items is undefined
const total = items.reduce((sum, item) => sum + item.price, 0);

// ✅ Good
const total = (items ?? []).reduce((sum, item) => sum + item.price, 0);

Interview Questions

▼

Q: What is the difference between var, let, and const?

var is function-scoped and hoisted (initialised to undefined). let and const are block-scoped and in the "temporal dead zone" until declared. const prevents reassignment but does not make objects immutable.

Q: Explain closures with an example.

function makeCounter() {
  let count = 0;
  return () => ++count;
}
const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
// count is preserved in the closure

Q: What is event delegation?

Instead of attaching a listener to every child element, you attach one listener to a parent element and use event.target to determine which child was clicked. This improves performance and works for dynamically added elements.

Q: What is the difference between null and undefined?

undefined means a variable has been declared but not assigned a value. null is an explicit assignment meaning "no value". Use null to intentionally clear a variable; you will usually encounter undefined when accessing missing properties.

Q: How does the JavaScript event loop work?

JavaScript is single-threaded. The event loop continuously checks the call stack and the task queue. When the stack is empty, it picks the next callback from the queue (macrotasks like setTimeout) or the microtask queue (like resolved Promises) and pushes it onto the stack.
